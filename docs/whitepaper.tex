\begin{abstract}
This whitepaper analyzes Mavi VPN, a Virtual Private Network architecture built primarily in Rust. It covers the IETF QUIC transport, the optional HTTP/2 CONNECT-IP transport over TLS/TCP, Encrypted Client Hello (ECH), MASQUE (RFC 9484), and the configurable MTU strategy. By using asynchronous I/O, dual-stack routing, and certificate pinning, Mavi VPN provides multiple transport choices for constrained network environments.
\end{abstract}

\section{Introduction and Core Philosophy}

Many VPN protocols expose recognizable wire patterns. WireGuard, for example, uses fixed-size handshake messages that a filtering system can fingerprint without decrypting their contents.

Mavi VPN discards these legacy assumptions. It operates on the principle that \textit{all} traffic is subject to hostile interception, active probing, and Deep Packet Inspection (DPI). Its QUIC censorship-resistant mode uses HTTP/3 framing and probe camouflage; its separate HTTP/2 mode provides a reliable TLS/TCP CONNECT-IP fallback without claiming to be ordinary browser traffic.

\subsection{Key Design Goals}
\begin{itemize}
    \item \textbf{Protocol Camouflage}: Using HTTP/3 application-layer framing (ALPN \texttt{h3}), a cover SNI, and desktop ECH GREASE based on RFC 9849. ECH uses HPKE as specified by RFC 9180.
    \item \textbf{Performance}: Using Rust's \texttt{tokio} asynchronous runtime, \texttt{bytes}-backed packet buffers, and hardware-accelerated cryptography through \texttt{aws-lc-rs}.
    \item \textbf{Network Resilience}: Supporting QUIC connection migration for mobile roaming and BBR congestion control for high-bandwidth or high-latency paths.
    \item \textbf{Memory Safety}: Rust prevents many buffer-overflow and use-after-free defects in safe code. Platform APIs, dependencies, configuration, and explicitly audited unsafe boundaries still require security review.
\end{itemize}

\section{System Architecture and Topology}

The ecosystem is separated into a backend, platform-specific privileged network services, and unprivileged user interfaces.

\subsection{Architectural Component Layout}
The architecture separates unprivileged UI tasks from highly privileged network stack manipulations using local Inter-Process Communication (IPC).

{\footnotesize
\begin{verbatim}
[ Graphical User Interface (Tauri) ] --------------+
  (Cross-Platform: Windows, Linux)                 |
                                             | (IPC: Unix socket / Named Pipe)
                                                   |
+-----------------------------------+              |
|   Platform Core (Windows/Linux)   | <------------+     [ Mobile Client ]
|   Privileged: WinTUN/Linux TUN    |                    (Android Kotlin UI)
|   Route & DNS Policy Management   |                             |
+-----------------------------------+                             | (JNI)
                 |                                                |
                 +----- QUIC Datagram or HTTP/2 Capsule Tunnel -----+
                 (TLS / MASQUE Connect-IP Capsules)
                                                   |
                                       +-----------------------+
                                       |      VPN Server       |
                                       |  (Rust + tokio + tun) |
                                       | Authentication & Pkt  |
                                       |     Routing Hub       |
                                       +-----------------------+
\end{verbatim}
}

\subsection{The Backend (Server-Side)}
The Linux-based backend is the routing hub. It terminates QUIC and optional HTTP/2 connections, verifies identity tokens or OpenID Connect (OIDC) JWTs, allocates private Dual-Stack IP addresses, and routes packets between the inner virtual network and the internet via Network Address Translation (NAT).

The server uses Rust's platform-default allocator; it does not configure a custom global allocator. Packet handling uses \texttt{bytes}-backed buffers and bounded asynchronous queues. Throughput and allocation behavior must be established with workload-specific benchmarks rather than allocator guarantees.

\subsection{Platform-Specific Clients}
Mavi VPN provides native drivers rather than relying on generic user-space wrappers:
\begin{itemize}
    \item \textbf{Windows (\texttt{windows} crate)}: Uses the \textbf{WinTUN} ring-buffer adapter. Route management is enacted via the Win32 API, and DNS policy is applied through the Windows \textbf{Name Resolution Policy Table (NRPT)} so matching queries use the tunnel resolver.
    \item \textbf{Linux (\texttt{linux} crate)}: Directly interacts with \texttt{/dev/net/tun} via low-level \texttt{ioctl} calls using the \texttt{tun} abstraction. It employs \texttt{systemd-resolved} interfacing to push the VPN DNS resolver to the top of the OS lookup table.
    \item \textbf{Android (\texttt{android} crate)}: Passes the Android \texttt{VpnService} file descriptor through JNI into the Rust asynchronous event loop via \texttt{tokio::io::unix::AsyncFd}. After setup, per-packet polling remains in the native Rust path.
\end{itemize}

\section{Transport \& Framing: The QUIC Paradigm}

Carrying a Layer 3 VPN over TCP can create nested retransmission and head-of-line blocking when inner TCP traffic experiences loss. Raw UDP, by contrast, leaves congestion control and encryption to the application protocol.

Mavi VPN leverages \textbf{QUIC} by default. We use the \texttt{quinn} Rust crate (implementation of IETF QUIC) to provide independent, unreliable datagrams in an encrypted, congestion-controlled wrapper. An optional HTTP/2 CONNECT-IP mode uses TLS/TCP and RFC 8441 Extended CONNECT with RFC 9297 capsules; its capsule delivery is reliable and ordered.

\subsection{Datagrams vs. Streams}
\begin{itemize}
    \item \textbf{The Data Plane (Datagrams)}: Encapsulated IP packets (IPv4/IPv6) are transmitted using QUIC Datagrams (RFC 9221). Datagrams are unordered and unreliable, so loss of one VPN packet does not block later packets on a QUIC stream; the inner protocol remains responsible for recovery.
    \item \textbf{The Control Plane (Streams)}: Configuration and authentication messages are sent over reliable, ordered QUIC streams before the client configures its virtual interface.
\end{itemize}

\subsection{HTTP/2 CONNECT-IP Fallback}
When \texttt{VPN\_HTTP2\_BIND\_ADDR} is configured, the backend exposes a separate
TLS/TCP listener that negotiates ALPN \texttt{h2}. Clients send an Extended CONNECT
request to \path{/.well-known/masque/ip/*/*/} with protocol \texttt{connect-ip}
and \texttt{capsule-protocol: ?1}. Address, route, configuration, reauthentication,
and IP packet data are exchanged as CONNECT-IP capsules inside HTTP/2 DATA frames.
HTTP/2 mode is mutually exclusive with QUIC HTTP/3 framing, censorship-resistant
mode, and ECH.

\subsection{Congestion Control: BBR (Bottleneck Bandwidth and RTT)}
Mavi VPN configures \textbf{BBR} congestion control for QUIC. BBR estimates bottleneck bandwidth and baseline RTT rather than reacting only to individual losses. Its effectiveness depends on the network path and must be measured under the intended workload.

\section{Censorship Resistance \& DPI Evasion}

Filtering systems can combine entropy analysis, handshake profiling, traffic shape, and active probes. Mavi VPN provides several countermeasures, but no camouflage mode can guarantee indistinguishability against every observer.

\subsection{Encrypted Client Hello (ECH --- RFC 9849; HPKE --- RFC 9180)}
In standard TLS 1.3, the Server Name Indication (SNI) is transmitted in plaintext, allowing firewalls to block connections to known VPN endpoints instantly.
\begin{itemize}
    \item Windows and Linux clients parse an administrator-provided hex \texttt{ECHConfigList}, use its \texttt{public\_name} as a cover SNI, and configure rustls \texttt{EchMode::Grease} through the \texttt{aws-lc-rs} HPKE implementation.
    \item Android parses the same \texttt{public\_name} and uses it as its SNI, but its \texttt{ring} crypto provider lacks HPKE and therefore does not emit an ECH extension.
    \item The backend generates and persists ECH config and key artifacts, but rustls server-side ECH acceptance is not wired in this release. It does not decrypt an inner ClientHello.
    \item Consequently, the current feature provides ClientHello camouflage and ECH compatibility testing, not full end-to-end ECH confidentiality.
\end{itemize}

\subsection{Active Probe Emulation (The Fake Nginx Server)}
If a DPI system (or a malicious actor) strips the ECH payload, guesses the endpoint, or initiates a raw scan on the UDP port, a standard VPN would drop the packet or issue a QUIC \texttt{CONNECTION\_CLOSE}, which immediately identifies it as a non-standard service.

Mavi VPN performs \textbf{Active Probe Emulation}. When an incoming connection fails authentication or presents an invalid ALPN, the backend does not silently drop it. Instead:
\begin{enumerate}
    \item A grace period of 50ms (\texttt{H3\_DETECTION\_GRACE}) is allocated to allow stream multiplexing.
    \item The engine shifts the connection context natively into an \texttt{h3} (HTTP/3) handler.
    \item It creates a control stream (\texttt{0x00 0x04 0x00}) representing HTTP/3 settings.
    \item It sends raw, QPACK-encoded headers equivalent to \texttt{:status 200} and \texttt{server: nginx}.
    \item It flushes an HTML layout representing the default "Welcome to nginx!" landing page directly onto the bidirectional QUIC stream.
\end{enumerate}
This behavior can reduce the value of simple active probes. Statistical or adaptive classifiers may still distinguish VPN traffic from ordinary browsing.

\subsection{MASQUE (RFC 9484) Connect-IP Encapsulation}
The "HTTP/3 Framing" flag uses an IETF-aligned MASQUE CONNECT-IP request and capsule data model.
\begin{enumerate}
    \item The client establishes a genuine HTTP/3 Request over a bidirectional QUIC stream: 
    \texttt{CONNECT /.well-known/masque/ip/*/*/ HTTP/3} with \texttt{:protocol=connect-ip}.
    \item Tunnel configuration uses HTTP/3 \textbf{Capsules}. Mavi uses \texttt{ADDRESS\_ASSIGN} (0x01) and \texttt{ROUTE\_ADVERTISEMENT} (0x03) to allocate IPs and advertise routes.
    \item A proprietary vendor capsule, \texttt{MAVI\_CONFIG} (0x4D56), transmits supplementary configuration (DNS parameters, Pinned MTU metadata).
    \item Data datagrams are prefixed with the RFC 9484 Quarter Stream ID and Context ID (\texttt{0x00 0x00}).
\end{enumerate}

\section{Authentication and Cryptography}

Pinned deployments do not rely solely on the public Web PKI trust store. A client accepts the configured leaf-certificate digest, so secure pin distribution and rotation are part of the deployment's trust model.

\subsection{SHA-256 Certificate Pinning}
\begin{itemize}
    \item During backend deployment, the server generates a TLS certificate and its private key.
    \item It immediately computes the SHA-256 fingerprint hash of the DER-encoded certificate and stores it in \texttt{cert\_pin.txt}.
    \item The client compares the leaf certificate's SHA-256 digest with one or more configured pins in constant time. A mismatch aborts the handshake with a certificate-pin error. This protects against an untrusted or compromised CA only when pin distribution and the client configuration remain trustworthy; it does not protect a compromised endpoint or stolen server key.
\end{itemize}

\subsection{Identity Provider Integration (Keycloak OIDC)}
While static tokens are suitable for private deployments, Mavi VPN's architectural scale targets enterprise environments requiring complex Identity and Access Management (IAM).
\begin{itemize}
    \item The platform integrates native \textbf{Proof Key for Code Exchange (PKCE) OAuth2} flows.
    \item The desktop client initializes a temporary loopback callback listener on \texttt{localhost:18923} and opens the machine's default browser at the Keycloak authorization endpoint.
    \item Upon successful SSO login, the IdP redirects back to the loopback listener with an authorization code.
    \item The VPN backend securely validates the resulting JSON Web Token against Keycloak's cryptographic JWKS (JSON Web Key Set) endpoint, enforcing \texttt{azp} (Authorized Party) checks in constant time. 
\end{itemize}

\section{The "Pinned MTU" Engineering Strategy}

MTU mismatch is a frequent source of VPN black holes. Exceeding the available path MTU can lead to dropped packets when discovery or fragmentation does not succeed.

\subsection{Evaluating Path MTU Discovery (PMTUD)}
QUIC can probe larger datagram sizes to discover the available path. Access technologies such as PPPoE or additional tunnels can reduce the usable MTU below Ethernet's typical 1500 bytes.
Fragmented UDP is not reliable across every network or filtering device, which can turn an oversized encapsulated packet into a silent failure.

\subsection{Dual-Boundary Pinned Limitation}
Mavi VPN bounds PMTUD variance with an operator-selected inner MTU in the
range 1280--1360. For QUIC transports the outer payload MTU is derived as
\texttt{VPN\_MTU + 80}; HTTP/2/TCP has no QUIC payload MTU.

\textbf{1. The Outer QUIC Payload: Derived from the Inner MTU}\\
We actively disable PMTUD within the Quinn stack:
\begin{lstlisting}
transport_config.mtu_discovery_config(None); 
transport_config.initial_mtu(vpn_mtu + 80);
transport_config.min_mtu(vpn_mtu + 80);
\end{lstlisting}
With the default \texttt{VPN\_MTU=1280}, the QUIC payload limit is 1360 bytes. The derived value keeps the encrypted UDP packet below common 1460-byte paths.

\textbf{2. The Inner Payload Tunnel: Configurable at 1280--1360 Bytes}\\
1280 Bytes is the minimum IPv6 link MTU. Configuring the virtual TUN adapter within the supported range tells the Windows, Linux, or Android network stack which maximum packet size the tunnel accepts.

\textbf{3. The Protocol Mathematics}
\begin{itemize}
    \item Default QUIC Payload: $1280 + 80 = 1360$ Bytes.
    \item Transport Overhead: $\sim$80 Bytes (40B for IPv6 + 8B for UDP + 32B for QUIC Headers/TLS Auth Tag).
    \item Available Payload: $\texttt{VPN\_MTU}$ Bytes.
    \item Max Inner Packet: the selected TUN MTU.
\end{itemize}
This sizing reduces fragmentation risk on common access networks, but it cannot guarantee a fragmentation-free path on every network.

\subsection{ICMP Asynchronous Emulation}
If the local stack still produces a packet larger than the configured TUN MTU, the Mavi VPN core intercepts it rather than forwarding it.
The core extracts the source/destination IPs and dynamically synthesizes an \textbf{ICMPv4/ICMPv6 "Packet Too Big" (PTB)} error payload directly inside the event loop, immediately injecting it back into the local operating system's ingress queue. This forces the host OS to accurately clamp its TCP Maximum Segment Size (MSS) without requiring messy server-side \texttt{iptables mangle} hacks.

\section{System Optimization and Kernel Offloading}

The implementation configures socket buffers and transport offload features to reduce overhead on high-throughput links.

\subsection{Socket Buffers and GSO}
Larger UDP socket buffers can absorb short bursts when the application or network stack cannot drain the queue immediately.
\begin{itemize}
    \item Mavi VPN requests 4~MB \texttt{SO\_SNDBUF} and \texttt{SO\_RCVBUF} socket buffers; the effective size remains operating-system dependent.
    \item \textbf{Generic Segmentation Offload (GSO)} is enabled in the transport configuration. When the operating system and network device support it, GSO can combine writes and reduce per-packet syscall overhead.
\end{itemize}

\subsection{IpGuard RAII and Dual-Stack Operations}
Server-side IP lease tracking uses RAII (Resource Acquisition Is Initialization). After authentication, an \texttt{IpGuard} owns the assigned IPv4 and IPv6 leases. When a session ends---through a clean disconnect, error, or timeout---dropping the guard returns those leases to the pool.
The network natively supports \textbf{Dual-Stack} deployments, assigning a dedicated \texttt{fd00::/64} IP and routing it accurately using \texttt{ip6tables} NAT66 masquerading to the WAN block.

\section{Conclusion}

The Mavi VPN architecture combines raw QUIC, HTTP/3/MASQUE camouflage, and an optional HTTP/2 CONNECT-IP fallback. Through careful packet validation, certificate pinning, authentication, and transport-specific MTU handling, Mavi VPN delivers high-performance edge routing while making the deployment trade-offs explicit: QUIC provides unreliable datagrams and roaming, while HTTP/2 provides reliable ordered capsules over TLS/TCP.
